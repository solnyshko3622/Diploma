# Use Case диаграмма для SQL Editor

## Описание системы
SQL Editor - это веб-приложение для создания, редактирования и выполнения SQL запросов с поддержкой проектов, аутентификации пользователей и интеграции с PostgreSQL через Taidon API.

## Актеры (Actors)

### 1. Неавторизованный пользователь (Guest User)
- Пользователь, который не вошел в систему
- Имеет ограниченный доступ к функциональности

### 2. Авторизованный пользователь (Authenticated User)  
- Пользователь, который прошел аутентификацию
- Имеет полный доступ к функциональности системы

### 3. Система Taidon API (External System)
- Внешняя система для выполнения SQL запросов
- Предоставляет PostgreSQL инстансы

## Use Cases

### Группа: Управление аутентификацией

#### UC-001: Регистрация пользователя
**Актер:** Неавторизованный пользователь  
**Описание:** Создание нового аккаунта в системе  
**Предусловия:** Пользователь не авторизован  
**Основной поток:**
1. Пользователь открывает модальное окно регистрации
2. Заполняет форму (имя, email, пароль, подтверждение пароля)
3. Система валидирует данные
4. Система создает новый аккаунт
5. Пользователь автоматически авторизуется

**Альтернативные потоки:**
- 3a. Данные невалидны - показать ошибки валидации
- 4a. Email уже существует - показать ошибку

#### UC-002: Вход в систему
**Актер:** Неавторизованный пользователь  
**Описание:** Аутентификация существующего пользователя  
**Предусловия:** Пользователь имеет аккаунт  
**Основной поток:**
1. Пользователь открывает модальное окно входа
2. Вводит email и пароль
3. Система проверяет учетные данные
4. Пользователь авторизуется в системе

**Альтернативные потоки:**
- 3a. Неверные учетные данные - показать ошибку

#### UC-003: Выход из системы
**Актер:** Авторизованный пользователь  
**Описание:** Завершение сессии пользователя  
**Основной поток:**
1. Пользователь выбирает выход из профиля
2. Система завершает сессию
3. Пользователь перенаправляется на страницу проектов

#### UC-004: Управление профилем
**Актер:** Авторизованный пользователь  
**Описание:** Просмотр и редактирование данных профиля  
**Основной поток:**
1. Пользователь открывает модальное окно профиля
2. Просматривает/редактирует данные профиля
3. Сохраняет изменения

### Группа: Управление проектами

#### UC-005: Просмотр списка проектов
**Актер:** Авторизованный пользователь  
**Описание:** Отображение всех доступных проектов пользователя  
**Основной поток:**
1. Пользователь переходит на страницу проектов
2. Система загружает список собственных и общих проектов
3. Отображается список проектов в виде карточек

**Альтернативные потоки:**
- 2a. Ошибка загрузки - показать сообщение об ошибке с кнопкой повтора

#### UC-006: Создание нового проекта
**Актер:** Авторизованный пользователь  
**Описание:** Создание нового проекта для работы с SQL  
**Основной поток:**
1. Пользователь нажимает "Новый проект"
2. Система создает проект с базовыми настройками
3. Создается первый SQL скрипт в проекте
4. Пользователь перенаправляется в редактор

#### UC-007: Открытие проекта
**Актер:** Авторизованный пользователь  
**Описание:** Открытие существующего проекта для редактирования  
**Основной поток:**
1. Пользователь выбирает проект из списка
2. Система загружает скрипты проекта
3. Открывается редактор с первым скриптом

### Группа: Управление SQL скриптами

#### UC-008: Создание нового скрипта
**Актер:** Авторизованный пользователь  
**Описание:** Создание нового SQL скрипта в проекте  
**Предусловия:** Проект открыт  
**Основной поток:**
1. Пользователь открывает модальное окно создания файла
2. Вводит название скрипта
3. Система создает новый скрипт с базовым содержимым
4. Скрипт становится активным в редакторе

#### UC-009: Редактирование скрипта
**Актер:** Авторизованный пользователь  
**Описание:** Изменение содержимого SQL скрипта  
**Предусловия:** Скрипт открыт в редакторе  
**Основной поток:**
1. Пользователь редактирует код в редакторе
2. Система выполняет синтаксическую подсветку
3. Выполняется локальная валидация SQL
4. Изменения автоматически сохраняются

**Расширения:**
- Подсветка ошибок синтаксиса
- Автодополнение SQL команд
- Автосохранение через 2 секунды после остановки печати

#### UC-010: Переименование скрипта
**Актер:** Авторизованный пользователь  
**Описание:** Изменение названия скрипта  
**Основной поток:**
1. Пользователь редактирует название в боковой панели
2. Система обновляет название в бэкенде
3. Название обновляется в интерфейсе

#### UC-011: Удаление скрипта
**Актер:** Авторизованный пользователь  
**Описание:** Удаление скрипта из проекта  
**Предусловия:** В проекте больше одного скрипта  
**Основной поток:**
1. Пользователь нажимает кнопку удаления скрипта
2. Система удаляет скрипт из бэкенда
3. Скрипт удаляется из списка
4. Если удаленный скрипт был активным, активируется первый доступный

### Группа: Выполнение SQL запросов

#### UC-012: Выполнение одного запроса
**Актер:** Авторизованный пользователь, Taidon API  
**Описание:** Выполнение активного SQL скрипта  
**Предусловия:** Скрипт содержит SQL код  
**Основной поток:**
1. Пользователь нажимает "Выполнить запрос" (F5)
2. Система проверяет доступность бэкенда
3. Подготавливается PostgreSQL инстанс (если нужно)
4. Запрос отправляется в Taidon API
5. Результаты отображаются в панели результатов

**Альтернативные потоки:**
- 2a. Бэкенд недоступен - показать ошибку
- 3a. Ошибка подготовки инстанса - показать ошибку
- 4a. Ошибка выполнения SQL - показать ошибку с подсветкой в редакторе

#### UC-013: Выполнение всех запросов
**Актер:** Авторизованный пользователь, Taidon API  
**Описание:** Последовательное выполнение всех скриптов проекта  
**Предусловия:** В проекте есть скрипты  
**Основной поток:**
1. Пользователь нажимает "Выполнить все" (Ctrl+F5)
2. Система запрашивает подтверждение
3. Подготавливается PostgreSQL инстанс
4. Все скрипты выполняются последовательно
5. Отображается сводка результатов

#### UC-014: Подготовка PostgreSQL инстанса
**Актер:** Система, Taidon API  
**Описание:** Создание и настройка PostgreSQL инстанса для выполнения запросов  
**Основной поток:**
1. Система проверяет наличие активного инстанса
2. Если инстанс отсутствует, создается новый через Taidon API
3. Инстанс настраивается и становится готовым к работе
4. Статус инстанса обновляется в интерфейсе

### Группа: Валидация и анализ кода

#### UC-015: Локальная валидация SQL
**Актер:** Система  
**Описание:** Проверка синтаксиса SQL на клиентской стороне  
**Основной поток:**
1. Пользователь вводит/изменяет SQL код
2. Система выполняет лексический анализ
3. Выполняется синтаксический анализ
4. Ошибки подсвечиваются в редакторе

#### UC-016: Бэкенд валидация SQL
**Актер:** Система, Taidon API  
**Описание:** Проверка SQL через PostgreSQL инстанс  
**Предусловия:** Бэкенд доступен, инстанс готов  
**Основной поток:**
1. Система отправляет код на валидацию в бэкенд
2. PostgreSQL проверяет синтаксис и семантику
3. Результаты валидации возвращаются
4. Ошибки отображаются в редакторе

### Группа: Настройки и конфигурация

#### UC-017: Изменение темы приложения
**Актер:** Пользователь  
**Описание:** Переключение между светлой и темной темой  
**Основной поток:**
1. Пользователь нажимает кнопку смены темы
2. Система переключает тему интерфейса
3. Настройка сохраняется локально

#### UC-018: Настройка редактора
**Актер:** Пользователь  
**Описание:** Изменение настроек редактора кода  
**Основной поток:**
1. Пользователь открывает меню настроек
2. Выбирает шрифт редактора
3. Выбирает тему редактора
4. Настройки применяются и сохраняются

#### UC-019: Управление валидацией
**Актер:** Авторизованный пользователь  
**Описание:** Включение/выключение бэкенд валидации  
**Предусловия:** Бэкенд доступен  
**Основной поток:**
1. Пользователь переключает настройку "Бэкенд валидация"
2. Система обновляет режим валидации
3. Настройка сохраняется

### Группа: Справочная информация

#### UC-020: Просмотр справочника SQL
**Актер:** Пользователь  
**Описание:** Получение справочной информации по SQL функциям  
**Основной поток:**
1. Пользователь открывает справочник SQL
2. Система отображает модальное окно со справочной информацией
3. Пользователь может искать и просматривать SQL функции

#### UC-021: Проверка подключения к бэкенду
**Актер:** Пользователь, Система  
**Описание:** Ручная проверка доступности Taidon API  
**Основной поток:**
1. Пользователь нажимает "Проверить подключение"
2. Система выполняет запрос к Taidon API
3. Проверяется статус PostgreSQL инстанса
4. Результат проверки отображается пользователю
